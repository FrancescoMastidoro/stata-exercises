## Background
The `nlsw88` dataset contains survey data on employed women in the United States drawn from the National Longitudinal Survey of Women, covering wages, hours, occupation, and demographic characteristics. In real labor economics research, raw survey data almost always contain missing values, inconsistently coded entries, and variables stored in unexpected types — identifying these problems before modeling is essential to avoid silently biased estimates and irreproducible results.

## Your tasks

1. **Audit the variable types** — Load the dataset with `sysuse nlsw88, clear` and run `describe`. Scan the storage type column and the value label column. Write a comment in your do-file listing: (a) which variables are stored as strings, (b) which variables are labeled numeric (i.e., numeric type with a non-empty value label), and (c) which variables are numeric with no label. Explain in one comment line why the distinction between (b) and (c) matters when you use an `if` condition to filter observations.

2. **Count and locate missing values** — Run `misstable summarize` to get a compact overview of missingness across all variables. Then pick the variable with the highest number of missing observations and run `codebook` on it. Write a comment noting: how many observations are missing, what share of the total sample that represents, and one plausible reason why that variable might be systematically missing in a survey of employed women.

3. **Flag and examine missing observations** — Using the variable you identified in Task 2, generate a new binary indicator variable called `flag_miss` that equals 1 when that variable is missing and 0 otherwise. Then run `tabulate flag_miss` to confirm the counts match what `misstable summarize` reported. Finally, run `summarize wage` separately for flagged and unflagged observations using `if` conditions and write a comment discussing whether missingness appears to be related to wage level.

4. **Check for impossible or suspicious values** — Run `summarize hours, detail` and inspect the minimum, maximum, and upper percentiles. Then run `summarize wage, detail` and do the same. Write a comment for each variable stating whether any values seem implausible given the real-world context (e.g., physiological limits on weekly hours, plausible wage ranges for 1988), and propose a concrete rule — expressed as an inequality you could use in an `if` condition — that would exclude the most suspicious observations.

5. *(Bonus)* **Cross-tabulate missingness with a categorical variable** — Use `tabulate occupation flag_miss` (the variable you flagged in Task 3) to examine whether missingness is concentrated in particular occupations. Compute the share of missing observations within each occupational category by adding the `, row` option. Write a comment identifying any occupation where the missing rate is notably higher than the overall missing rate you found in Task 2, and discuss what that pattern might imply for an occupational wage analysis.

- **Hint 1**: For Task 2, `misstable summarize` prints a table only for variables that have at least one missing value, showing the number and percentage missing. If you want to verify a specific variable's missingness by hand, you can also run `count if varname == .` — but recall that the missing value code for strings is `""` (empty string), not `.`, so you need to know the variable's type before writing that condition.
- **Hint 2**: For Task 3, to generate a binary flag for missing values you can use `generate flag_miss = missing(varname)`, which returns 1 when the value is missing and 0 otherwise — this works correctly for both numeric and string variables, so it is safer than writing `== .` directly.

> **Fresh hint**: A common mistake is writing `if region == "Europe"` — Stata will reject this because `region` is a labeled numeric variable, not a string; the labels are only display decoration. The actual filter must use the integer code, for example `if region == 2`. If you are unsure which integer maps to which label, run `label list` after `describe` to find the name of the value label scheme attached to `region`, then run `label list schemename` to print the full code-to-label mapping.
